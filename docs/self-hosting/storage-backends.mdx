# Storage Backends

Sifter supports three storage backends, selected via `SIFTER_STORAGE_BACKEND`.

## Filesystem (default)

```bash
SIFTER_STORAGE_BACKEND=filesystem
SIFTER_STORAGE_PATH=./uploads
```

Files are stored on the local filesystem. For Docker deployments, mount `SIFTER_STORAGE_PATH` to a persistent volume.

## S3-compatible

Works with AWS S3, MinIO, Cloudflare R2, Backblaze B2, and any S3-compatible provider.

```bash
SIFTER_STORAGE_BACKEND=s3
SIFTER_S3_BUCKET=my-sifter-bucket
SIFTER_S3_REGION=us-east-1
SIFTER_S3_ACCESS_KEY_ID=AKIA...
SIFTER_S3_SECRET_ACCESS_KEY=...
```

For MinIO or other S3-compatible services, set the endpoint:

```bash
SIFTER_S3_ENDPOINT_URL=http://minio:9000
```

## Google Cloud Storage

```bash
SIFTER_STORAGE_BACKEND=gcs
SIFTER_GCS_BUCKET=my-sifter-bucket
SIFTER_GCS_PROJECT=my-gcp-project
# Leave SIFTER_GCS_CREDENTIALS_FILE empty to use Application Default Credentials
# Or point to a service account key file:
SIFTER_GCS_CREDENTIALS_FILE=/secrets/gcs-key.json
```

## Notes

- Documents are downloaded to a temp file before extraction, regardless of backend
- The backend is abstracted behind a `StorageBackend` protocol — adding custom backends is straightforward
